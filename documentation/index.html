Documentation Example & workflow

Start the script: 

$ ./sitespeed.io http://peterhedenskog.com

The script will log what is happening, it will look something like this: 

$ Will crawl from start point http://peterhedenskog.com with depth 1 ... this can take a while
$ Will start fetching all a links ...
$ http://peterhedenskog.com/
$ http://peterhedenskog.com/category/work/
$ ...
$ Fetched 5 pages
$ Analyzing http://peterhedenskog.com/
$ ...
$ Create individual pages
$ Create the pages.html
$ Create the summary: index.html
$ Finished, see the report sitespeed-result/sitespeed-peterhedenskog.com-2012-08-15-20-18-48/index.html
$

It will create the following directory structure (the first directory is named after the site you are testing and when the test was executed): 

sitespeed-result
|
`--sitespeed-peterhedenskog.com-2012-08-15-20-18-48
   |
   | -- css
   |
   | -- data
   |
   | -- img
   |
   | -- js
   |
   | -- pages
   |
   | -- pages.html
   | -- pages.png
   | -- summary.png
   ` -- index.html

The data directory hold all xml files that's the result from the analyze (the raw data). Pages hold the result (as HTML) for all unique page tests. The pages.html is all pages data, summarized in one page. The index.html is the summary page, holding information on a high level.

The workflow is like this:

Site summary page

Sitespeed.io creates three different kind of outputs. The summary page is kind of executive summary page where green is good, yellow means that you should look into it and red is bad, sometimes really bad. The page show a summary of some best practices for good web performance.

Pages summary page

The pages page, show data collected for each page. At the moment the following data is showed: Number of javascript/ccs/images/cssimages & font files, total requests, total requests with a primed cache, the size of the page, the total size of the page (including all assets), total size with primed cache, the load time of the page, number of single point of failures, number of javascript loaded synchronously in head, the sitespeedscore and a direct link to test the page at http://webpagetest.org . Click the magnifier to see the response header of the page, and click the url, to see all the specific collected page data.

Page output

Every page that is tested, also get a corresponding html page, where (almost) all data is shown. You will see all Yslow rules that are below 100, all assets on the page together with the response headers for that asset (click on the magnifier).
